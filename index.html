Heisann!

Det du glaner på nå er en klient.

Ikke for å skryte, men jeg er laget kun med nettleserkode -- HTML og JavaScript -- og kan kjøre hvorsomhelst på hvasomhelst ;-)

Jeg er interaktiv og full av eksempler, informasjon og pekere som kan være nyttig for å lære om Dataporten for første gang eller bruke som referansepunkt lenger ned i løypa.

Om du vil kjøre meg på din egen server kan du laste meg ned på GitHub og registrere meg på nytt i Dataporten som din egen klient. Les videre så skal jeg vise deg hvordan du gjør det.

Denne demoen antar at du er autentisert med Feide. Om du velger noe annet (eks. Facebook eller Twitter) så er det også greit, men se da bort fra referanser til Feide-relaterte ting :)

Klienter og API Gatekeepere

Klienter

I jomfruturen din erfarte du hvordan vi snakket med Dataporten for identifisering og uthenting av informasjon. La oss nå se kjapt på hvordan dette går for seg i bakgrunnen:

API Gatekeepere (API GK)

Frem til nå har vi snakket mest om autentiserings-aspektet ved Dataporten og noen kjerne-endepunkter som hun tilbyr, men dette er ikke alt.

En super-viktig og super-nyttig funksjonalitet du får med Dataporten er muligheten til å bruke/dele informasjon via hennes API Gatekeeper. Her er det så mye kult jeg gjerne vil snakke om for å virkelig få deg til å se mulighetene, men min funksjon er å tilby en "liten UNINETT Dataporten demonstrasjon" så jeg skal forsøke å fatte meg i korthet.

La meg først likevel si at API GK-aspektet tilfører, som jeg ser det, to tre fire fem hovedgevinster:

  1. Den gjør det lett for deg å dele datakilder (med andre eller bare dine egne klienter) på en sikker måte (heter ikke gatekeeper for ingenting)
  2. Den gir deg mulighet til å "tappe" inn i og sy sammen data fra Dataporten samt mange ulike APIer (dine egne og andres) - en "fleretasjers databuss" om du vil
  3. API GK muliggjør tilgangsstyring på flere nivåer: hvilke klienter, hvilke scopes, brukere...
  4. Mindre admin: Dataporten Dashboard er selvbetjent, og tjenestetilbydere kan abonnere på dine APIer uten at du trenger å gjøre noe. Dashboard gir deg også innsikt i hvem som bruker dine data.
  5. Selv om ditt endepunkt flyttes forblir API GK URL konstant (trenger bare oppdatere i Dashboard)

Sjekk ut dokumentasjon og enkel kode for vitse-API på GitHub (motivator: du finner ALLE de gode vitsene i kildefila!!).

Hvordan jeg er skrudd sammen

Jeg nevnte innledningsvis at jeg er skrevet i kun HTML og JavaScript. Mens tradisjonell Feide-pålogging krever et relativt komplekst oppsett på serverside med en SAML 2.0 programvarepakke (f.eks. SimpleSAMLphp), kan du altså med Dataporten autentisere brukere med bare JavaScript om du vil. Åja, og så slipper du selvfølgelig styret med idp-test, XML metadata, varsling til Feide om hvilke vertsorganisasjoner som skal kunne bruke tjenesten og at vertsorganisasjonene må aktivt åpne opp for at deres brukere kan ta i bruk tjenesten, osv...

Selv bruker jeg Andreas Solberg sin JSO-modul for OAuth 2.0 (GitHub) for å ta seg av autentisering. Kode for å få til dette er sørgelig enkelt, se her da: 


// Inkluder JSO-modul
<script type="text/javascript" src="js/auth/jso.js">
// Konfigurer JSO 
var min_jso_instans = new JSO({
client_id : 'id fra Dataporten Dashboard',
redirect_uri : 'URL til der jeg bor (gjerne 127.0.0.1 mens du utvikler)',
authorization	: 'https://auth.dataporten.no/oauth/authorization'
});
// callback() sender deg til Dataporten og fanger hennes respons ved retur. callback() bør kalles så tidlig som mulig, før applikasjonen lastes. 
min_jso_instans.callback();

Når bruker så er autentisert kan du bruke JSO som følger: 


// Slå på JQuery for JSO (forenkler AJAX kall)
JSO.enablejQuery($);
// Gjør et kall med din JSO instans - 'min_jso_instans'
min_jso_instans.ajax({
url: 'https://auth.dataporten.no/userinfo',
dataType: 'json'
}).done(function(userObj){
// Voila!
});
// Utlogging gjør jeg på denne måten:
min_jso_instans.wipeTokens();				
window.location.replace("https://auth.dataporten.no/logout");

...du kan lese mer om - og laste ned - JSO på GitHub. Du kan også studere kildekoden min for å se hvordan jeg gjør det.

En liten ting til: klienter må ikke bruke JSO (eller JavaScript) for å autentisere seg med Dataporten altså. Standarden heter OAuth 2.0, og du kan bruke nesten hva som helst for å snakke med den. Her er en haug med biblioteker i mange ulike språk (Python, PhP, Ruby, .net, Java +++++) som du kan se på, eller du kan skrive din egen. Det er også lett å modifisere moduler som er ferdigskrevet for autentisering med Google/Facebook/Twitter/etc til å virke med Dataporten.

Felles for alle biblioteker/moduler er at de følger en standard som egentlig er veldig enkel (når du først har skjønt flyten) og frittstående i natur.

Hvordan lage og dele sitt eget API

La oss studere det supermorsomme vitse-api'et i litt mer detalj. For mer utfyllende informasjon kan du forresten laste ned/se på kildekode på GitHub.

APIet er et enkelt PHP-script som ligger på en webtjener og alt det gjør er å ta imot en henvendelse og sende et tilfeldig svar (en vits) tilbake. Scriptet forventer at henvendelser kommer fra en Dataporten-klient via Dataporten GK på følgende URL:
https://simons-vitser.gk.dataporten.no/api/connect/simons-vitser/

APIet ligger egentlig på et helt annet domene, men med Dataporten rutes altså alle kall via gatekeeper på URLen over. Å registrere et API i Dataporten Dashboard er like enkelt som å registrere en klient; tilfør en identifikator, et navn og en URL og vips!

SSL: Dataporten GK vil bare snakke med ditt API dersom tjeneren din benytter https med et godkjent sertifikat ('self-signed' er ikke ok).

La oss si at vitse-APIet egentlig lå her: https://api.eksempel.no/api/connect/simons-vitser/index.php.

I Dataporten Dashboard oppgir vi da:

  • endpoint : https://api.eksempel.no
  • identifikator : simons-vitser

URLen som Dataporten-klienter må bruke for å snakke med APIet ditt blir da:
https://simons-vitser.gk.dataporten.no/api/connect/simons-vitser/

Headere fra Dataporten GK

For å sikre oss om at klienten som kaller APIet faktisk er en autentisert Dataporten-klient, sender Dataporten GK med flere nyttige headere. I PHP kan vi titte på f.eks. $_SERVER arrayen for å finne ut mer. Dersom klientkallet kom via Dataporten GK kan du forvente at disse feltene medfølger: 


// Min ID som Dataporten ga meg ved registrering
"HTTP_X_dataporten_CLIENTID": "f299e692-473a-4ac1-91c3-7e7872d1548d",
// Scopes jeg har tilgang til
"HTTP_X_dataporten_SCOPES": "9,15",
// Bruker-ID til han som besøkte meg
"HTTP_X_dataporten_USERID": "835c8888-08a9-4332-9a18-074944355ad8",
// Feide ID for samme bruker
"HTTP_X_dataporten_USERID_SEC": "feide:simon@uninett.no", 

// API GK bruker/pass ("Trust"-meny i Dashboard for ditt API.)
"PHP_AUTH_USER": "dataporten",
"PHP_AUTH_PW": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",

Med informasjonen over kan vi altså:

  • Autentisere klienten (sjekk at GK sender med riktig bruker/passord)
  • Si noe om klienten som er på besøk (kjenne igjen (ID) og vite tilganger (scopes))
  • Si noe om bruker som besøker klienten (ID, Feide)
